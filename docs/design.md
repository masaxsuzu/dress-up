# 自分専用ファッションアプリ 設計サマリ

## 1. 全体ビジョン（4機能）

1. 自分用アバター（体型反映の汎用キャラ）に服を着せ替え
1. 写真ベースのバーチャル試着
1. スタイル提案
1. 写真/テキストから手持ち服を登録

## 2. MVPスコープ

### 採用：「手持ち服の登録」一機能のみ

**フロー**: 撮影 → VLM で属性抽出 → D1 に保存 → 一覧・検索

### 選定理由

- 他3機能すべての土台になる（着せ替えに服がない・提案する服がない、では動かない）
- VLMの精度・出力スキーマを実データで検証できる。ここでコケると後段の設計が崩れる
- 単体でも「デジタルクローゼット」として実用的
- 1〜2週末で形になる規模

## 3. 技術スタック（Cloudflare完結）

|層      |採用                                  |メモ                       |
|-------|------------------------------------|-------------------------|
|フロント   |Next.js on Cloudflare Pages         |App Router               |
|バックエンド |Pages Functions                     |TypeScript               |
|DB     |D1                                  |SQLiteベース。普段使いの感覚に近い     |
|画像ストレージ|R2                                  |egress無料                 |
|認証     |Cloudflare Access                   |Zero Trust 無料プラン（〜50ユーザー）|
|VLM    |Claude Haiku 4.5 or Gemini 2.5 Flash|コスト・精度どちらも実用域            |

### 認証戦略

- Cloudflare Access で Google/GitHub ログインを設定
- メールアドレス allowlist に自分だけ登録
- **認証コードを1行も書かずに「自分しか入れないアプリ」が完成**

## 4. MVPで割り切ること

- **背景除去**: 白背景・平置き撮影で代用（`rembg` は Workers で動かない）
- **アバター/試着/提案**: フェーズ2以降に先送り
- **Streamlit**: 不採用（Workers上のPythonは未成熟。TypeScriptで書く）
- **スキーマ後方互換**: 取らない。破壊的変更時は写真を撮り直す

## 5. コスト

- Cloudflare 一式: **0円**（Pages/D1/R2/Access すべて無料枠内）
- VLM API: 服100着登録しても**数十円**レベル（Haiku 4.5 vision）
- ドメイン: 任意

## 6. フェーズ2以降の宿題

|機能      |候補手段                                    |
|--------|----------------------------------------|
|バーチャル試着 |IDM-VTON / Kolors を Replicate 経由        |
|体型反映アバター|Ready Player Me など既存サービス検討              |
|スタイル提案  |蓄積した服データ + LLM                          |
|背景除去    |Workers AI のセグメンテーションモデル / remove.bg API|

## 7. 次の判断ポイント

1. **VLM抽出スキーマ設計** — 服のメタデータをどう構造化するか（カテゴリ / 色 / 素材 / シルエット / シーズン / フォーマル度 / タグ等）。アプリ全体の表現力を決める核心
1. **画面フロー設計** — 撮る → 確認・編集 → 保存 の最低限UI

## 8. オープン課題・備忘

- D1 のサイズ上限と画像メタデータ量の見積もり
- R2 の画像命名規約（オリジナル/サムネイル）
- VLM 抽出失敗時のフォールバック（手動編集前提か、再生成か）
- Cloudflare Access のセッション有効期限設定
